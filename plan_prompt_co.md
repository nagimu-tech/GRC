# План-промпт: веб-приложение учёта выпускников психологических курсов

Документ — самодостаточное техническое задание для реализации. Все ключевые решения зафиксированы; помеченные «(по умолчанию)» можно при желании поменять, но они не блокируют старт.

---

## 1. Суть и роли

CRM учёта участников психологических курсов с **изоляцией данных между компаниями** (multi-tenant) и двумя режимами работы.

Приоритет режимов:

1. **Главный режим — учёт** студентов, ассистентов и тренеров и их участия в запусках курсов (кто, когда, в какой роли, в дальнейшем — кто сколько оплатил). Это ядро системы; вокруг него строится модель данных.
2. **Вспомогательный режим — прозвон** перед встречей выпускников. Узкий мобильный сценарий поверх тех же данных.

Роли пользователей приложения:


| Роль                                           | Что может                                                                                                                                                                  |
| ---------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Системный администратор** (Django superuser) | Всё во всех компаниях; создаёт компании и администраторов компаний; видит кандидатов на межкомпанийное сопоставление людей и выполняет перепривязку                        |
| **Администратор компании**                     | Полный CRUD в пределах своей компании; создаёт и блокирует прозвонщиков своей компании; не видит чужие компании и не видит межкомпанийные совпадения людей                 |
| **Прозвонщик**                                 | В пределах своей компании: видит списки курсов и выпускников; редактирует только отметки прозвона (статус прихода + комментарий) для активной встречи; ПДн людей не правит |

## 2. Стек (выбран)

- **Backend:** Django 5.2 LTS, Python 3.12+; практичный Docker-образ по умолчанию — `python:3.12-slim`.
- **БД:** PostgreSQL 16, в Docker Compose закрепить образ `postgres:16`; драйвер БД — `psycopg` 3.
- **Интерактивность:** HTMX (инлайн-сохранение отметок без перезагрузки) + точечно Alpine.js.
- **Стиль:** Tailwind CSS (mobile-first для режима прозвона).
- **Язык интерфейса:** весь видимый UI на русском языке: меню, формы, кнопки, статусы, ошибки, пустые состояния, подсказки, экспортные заголовки и административные экраны. Внутренние enum/code constants могут оставаться на английском.
- **Полный админ-режим:** собственные CRUD-вью на базе Django CBV (`ListView/CreateView/UpdateView/DeleteView`) + `ModelForm` — для полного контроля над изоляцией и ролями. Стандартная Django Admin включается только для системного администратора как back-office.
- **Аутентификация:** встроенная Django auth, кастомная модель пользователя, хеширование Argon2 (`argon2-cffi`).
- **Доп. библиотеки:** `django-simple-history` (аудит), `django-axes` (защита логина), `whitenoise` (статика), опционально `django-otp` (2FA для админов).
- **Очереди/фоновые задачи:** для первой версии Celery/Redis не нужны; текущие сценарии закрываются синхронным Django + HTMX + PostgreSQL.

## 3. Мультитенантность (изоляция компаний)

Однобазовая построчная изоляция. Компания = тенант.

- Базовый класс `TenantModel` с FK `company` и менеджером `TenantManager`, который **обязан** фильтровать по активной компании.
- `company` FK дублируется (денормализуется) на **каждой** тенант-модели, включая дочерние, — чтобы любая выборка фильтровалась одним полем без джойнов и без риска утечки.
- Middleware резолвит `request.company` из `request.user.company`.
- Миксин для вью `TenantScopedMixin`: автоматически фильтрует `get_queryset()` по `request.company` и проставляет `company` при создании.
- Системный администратор обходит фильтр / переключает активную компанию.
- **Тесты на изоляцию обязательны:** пользователь компании А не должен получить ни одной записи компании Б ни через список, ни через прямой URL по id.

`Person` — глобальная запись личности без `company`. Изоляция для рабочих экранов обеспечивается через `CompanyPerson`: привязку человека к конкретной компании. Администраторы и прозвонщики компании всегда работают с `CompanyPerson`, `Participation` и `CallRecord`, а не с глобальным списком `Person` напрямую.

Один и тот же человек может быть студентом, ассистентом или тренером в разных компаниях без дублирования глобальной личности. Если при создании записи обнаруживается, что такой человек уже есть в другой компании, системный администратор может выполнить **перепривязку** `CompanyPerson` к существующему `Person`. Администраторы компаний не видят чужие компании и не получают информацию о межкомпанийных совпадениях.

## 4. Модель данных

> Тренеры — НЕ отдельная сущность. Тренер, ассистент и студент — это **роль в записи участия**. Так человек накапливает историю всех ролей по времени.

### Company (тенант)

`name`, `is_active`, контактные данные, `created_at`.

### User (extends `AbstractUser`)

`company` (FK, null у системного админа), `role` (SYSTEM_ADMIN / COMPANY_ADMIN / CALLER) + стандартные поля. Кастомная модель определяется **до первой миграции**.

### Position (Должность) — справочник, **глобальный (общий для всех компаний)**

`name`, `description`, `is_active`, `order`.
**НЕ** тенант-модель: не наследует `TenantModel`, не имеет `company`, не фильтруется по компании. Один общий список должностей виден и используется во всех компаниях; редактируется только системным администратором. Обслуживает выбор должности в записи участия (см. Participation).

### Course (Курс / программа) — company-scoped

`company`, `name`, `description`, `is_active`.
Это программа, а не конкретный запуск.

### CourseSession (Запуск курса / занятие) — company-scoped (через Course)

`course`, `company`, `label`, `start_date`, `end_date`, `location_format`, `notes`.
Источник данных «когда проходил». Один курс имеет много запусков.

### Person (Человек — глобальная запись личности) — global-scoped

`last_name`, `first_name`, `middle_name` (ФИО раздельно — для сортировки и поиска), `birth_date`, `phone`, `email` *(опц.)*, `created_at`, `updated_at`.

`Person` не наследует `TenantModel`, не имеет `company` и не используется напрямую в рабочих экранах компании. Это каноническая запись личности, которая позволяет не дублировать одного и того же человека при участии в разных компаниях. Прямой просмотр глобального списка, межкомпанийные совпадения и перепривязки доступны только системному администратору.

Если один `Person` уже привязан к нескольким компаниям, изменение его идентифицирующих полей (`ФИО`, дата рождения, телефон, email) должно быть доступно только системному администратору либо отдельным безопасным сценарием подтверждения, чтобы правка в одной компании не изменила неожиданно данные для другой.

### CompanyPerson (Человек в компании / привязка к компании) — company-scoped

`company`, `person` (FK → Person), `notes`, `consent_stored` (согласие на хранение ПДн), `consent_contact` (согласие на обзвон), `is_active`, `created_at`.

Именно `CompanyPerson` показывается в разделе «Люди» внутри компании и используется во всех company-scoped связях. Для пользователя компании это выглядит как обычная карточка человека, но технически карточка отделена от глобальной личности.

Мягкий контроль дублей и сопоставление при создании:

- Внутри текущей компании показывать предупреждения при совпадении телефона, полного ФИО или ФИО + даты рождения среди существующих `CompanyPerson`.
- Между компаниями совпадения по ФИО, ФИО + телефону и ФИО + дате рождения не раскрывать администраторам компаний. Такие совпадения попадают в очередь/экран системного администратора как кандидаты на перепривязку.
- Жёстких уникальных ограничений по ФИО или телефону не делать: возможны однофамильцы, смена номера, ошибки ввода и неполные данные.
- Перепривязка выполняется только системным администратором: `CompanyPerson.person` меняется с ошибочно созданного `Person` на уже существующий `Person`; история участий и прозвонов остаётся на `CompanyPerson` и не теряется.

### Participation (Участие) — company-scoped — ядро системы

Связывает `CompanyPerson` ↔ `CourseSession` с указанием роли. Один человек в компании → много участий.

- `company` (денорм), `company_person` (FK → CompanyPerson), `session` (FK → CourseSession)
- `role`: **STUDENT / ASSISTANT / TRAINER**
- `chosen_position` (FK → Position, null) — «какую должность выбирал» / должность в контексте участия
- `notes`, `created_at`

Тренеры запуска = 1–3 записи `Participation(role=TRAINER)`. Прогресс «студент → ассистент → тренер» в рамках компании виден как набор участий одного `CompanyPerson` с разными ролями по датам сессий. Системный администратор при необходимости может увидеть сводную историю одного глобального `Person` по нескольким компаниям; пользователи компании видят только историю своей компании.

**Планируемое расширение (оплата участия).** На `Participation` позже добавляются поля оплаты, зависящие от роли: для студента — тип участия (полное / со скидкой) и сумма; для ассистента — ассистентский сбор. Ориентировочно: `payment_type` (FULL / DISCOUNTED / ASSISTANT_FEE / FREE), `amount`, `currency`, `paid_at`, `payment_note`. Заложить структуру сразу необязательно, но модель `Participation` спроектировать с расчётом на безболезненное добавление этих полей (учёт оплат — это атрибут конкретного участия, а не человека).

### Event (Встреча выпускников / прозвон) — company-scoped

`company`, `course` (FK, null — встреча может быть по конкретному курсу или общей), `title`, `date`, `location`, `is_active`, `notes`.
То, против чего ведётся прозвон.

- `assigned_callers` (M2M → User) — ассистенты, назначенные на этот прозвон. Доступ к прозвону имеют только назначенные пользователи (плюс администратор). Наборы людей между ассистентами **не** разбиваются заранее — это общий пул с механизмом захвата (см. CallRecord).

### CallRecord (Отметка прозвона) — company-scoped

`company`, `event` (FK), `company_person` (FK → CompanyPerson), `status`, `comment`, `called_by` (FK User), `claimed_by` (FK User, null), `claimed_at` (datetime, null), `updated_at`.
`status`: NOT_CALLED *(дефолт)* / WILL_COME / WONT_COME / MAYBE / NO_ANSWER.
Уникальность `(event, company_person)`. История сохраняется естественно: у каждой встречи свой набор отметок.

**Защита от двойного обзвона (захват записи).** В рамках одного прозвона человек, которого уже взяли или которому уже звонили, недоступен другим ассистентам:

- Доступен к набору = `claimed_by IS NULL` **и** `status = NOT_CALLED`.
- Когда ассистент берёт человека (открывает/начинает звонок), запись атомарно захватывается: `claimed_by` ставится условным `UPDATE ... WHERE claimed_by IS NULL` (или через `select_for_update`), чтобы при гонке выиграл ровно один ассистент. После этого человек исчезает из доступного пула у остальных.
- После звонка проставляется `status` и `called_by`; запись остаётся закреплённой и больше не выдаётся другим.
- Опционально: тайм-аут захвата — если звонок не завершён за N минут, запись освобождается обратно в пул. На старте можно не реализовывать.
- Записи `CallRecord` создаются заранее на всех подходящих людей события (дефолт NOT_CALLED, без захвата) либо создаются в момент захвата — тогда роль замка играет уникальное ограничение `(event, company_person)` (первый INSERT выигрывает).

## 5. Вспомогательный режим прозвона (mobile-first)

Узкий мобильный сценарий поверх данных учёта (не главный режим). Доступ — только ассистентам, назначенным на этот прозвон (`Event.assigned_callers`), и администратору.

Поток: активный **прозвон (Event)** → фильтр по **Курсу** → общий пул **студентов** этого курса → ассистент берёт человека из пула → звонит → отмечает результат.

Экран (одна колонка, крупные элементы):

1. Активный прозвон (предвыбран среди назначенных пользователю), фильтр по курсу.
2. Поиск по имени; по умолчанию показывается **доступный пул** (ещё не взятые и не обзвоненные); переключатель «мои отмеченные».
3. Строка списка: ФИО · даты прохождения (из сессии) · телефон как ссылка `tel:` (на смартфоне открывает набор) · переключатель статуса (бейджи) · поле комментария.
4. Взятие человека и любое изменение статуса/комментария → HTMX-POST с атомарным захватом `CallRecord`, мгновенный отклик; человек уходит из доступного пула у других ассистентов в реальном времени (без двойного обзвона).
5. Пагинация / ленивая подгрузка для длинных списков.
6. Пул людей = у кого есть `Participation(role=STUDENT)` в любой сессии выбранного курса и чья запись `CallRecord` ещё доступна.

Статусы — цветными бейджами для быстрого визуального сканирования. Все подписи, статусы и сообщения в режиме прозвона отображаются на русском языке.

## 6. Полный админ-режим (экраны)

- **Дашборд** (сводка по компании).
- **Компании** — только системный администратор.
- **Пользователи** — администратор компании создаёт/блокирует прозвонщиков своей компании.
- **Справочник должностей** — CRUD; глобальный, общий для всех компаний, редактирует только системный администратор. Администратору компании виден только на чтение (при выборе должности в участии).
- **Курсы** — CRUD.
- **Запуски курса** — CRUD; в форме запуска инлайн-назначение 2–3 тренеров (выбор существующего `CompanyPerson` или создание нового) = создание `Participation(role=TRAINER)`; так же добавление студентов/ассистентов.
- **Люди** — CRUD по `CompanyPerson`, поиск, локальный контроль дублей внутри компании.
- **Сопоставление людей** — только системный администратор: список кандидатов на межкомпанийное совпадение, сравнение ФИО/телефона/даты рождения, перепривязка `CompanyPerson` к существующему `Person`.
- **Карточка человека в компании** — таймлайн всей истории участий (студент/ассистент/тренер) по датам в рамках текущей компании — наглядно показывает прогресс.
- **Глобальная карточка человека** — только системный администратор: сводка привязок `CompanyPerson` и истории по компаниям без нарушения прав обычных пользователей.
- **Участия** — управляются преимущественно из карточки человека и формы запуска.
- **Встречи / прозвоны** — CRUD, флаг активного прозвона; назначение ассистентов на конкретный прозвон (`assigned_callers`).
- **Экспорт** списков (CSV/Excel) — опционально; заголовки и значения статусов в экспорте на русском языке.

## 7. Безопасность и приватность

Данные о прохождении психологических курсов — чувствительные.

- HTTPS only, HSTS, `Secure`/`HttpOnly`/`SameSite` cookies, CSRF, защита от XSS — захардить дефолты Django.
- Argon2 для паролей; `django-axes` против перебора; 2FA для админов — опционально.
- **Проверка роли и тенанта на сервере на каждом действии** (не только скрытие в шаблоне). Изоляция на уровне менеджера запросов — defense in depth.
- `Person` глобален, поэтому прямой доступ к нему запрещён для администраторов компаний и прозвонщиков. Рабочие запросы компаний всегда идут через `CompanyPerson` с фильтром по `company`.
- Межкомпанийные совпадения людей не раскрываются пользователям компаний. Очередь сопоставления, сравнение записей и перепривязка доступны только системному администратору.
- Аудит (`django-simple-history`): кто менял ПДн; кто выполнял перепривязку `CompanyPerson`; журнал доступа к спискам прозвона.
- Минимизация данных, флаги согласия на `CompanyPerson`, право на удаление/деактивацию человека в компании, политика хранения/ретенции.
- Шифрованные бэкапы БД, off-site.
- **Юридическое (не юр. консультация):** проработать со специалистом правовое основание обработки (GDPR; при данных граждан РФ — 152-ФЗ), согласия людей на хранение и обзвон, ответственного за обработку. Технически приложение это поддерживает (согласия, удаление/деактивация, аудит).

## 8. Развёртывание

Рекомендуемый вариант для Beget: **Beget Cloud VPS + Docker/Docker Compose**, а не обычный виртуальный хостинг и не one-click Django-образ.

Причины:

- У Beget есть VPS-образ Docker на Ubuntu 24.04 с Docker CE и Docker Compose: [https://beget.com/en/cloud/marketplace/docker](https://beget.com/en/cloud/marketplace/docker)
- Обычный виртуальный хостинг Beget для Python/Django больше похож на ручной virtualenv/passenger-сценарий; для новых версий Python может потребоваться ручная сборка Python/OpenSSL: [https://beget.com/ru/kb/how-to/web-apps/python](https://beget.com/ru/kb/how-to/web-apps/python)
- One-click Django-образ Beget удобен для старта, но в описании указан Python 3.10.6, что не соответствует требованию Python 3.12+: [https://beget.com/en/cloud/marketplace/django](https://beget.com/en/cloud/marketplace/django)
- Django 5.2 LTS официально поддерживает Python 3.12 и PostgreSQL 14+, поэтому PostgreSQL 16 подходит: [https://docs.djangoproject.com/en/5.2/releases/5.2/](https://docs.djangoproject.com/en/5.2/releases/5.2/) и [https://docs.djangoproject.com/en/5.2/ref/databases/](https://docs.djangoproject.com/en/5.2/ref/databases/)

Docker Compose:

- `web` — Django 5.2 LTS + gunicorn на базе `python:3.12-slim`.
- `db` — PostgreSQL `postgres:16` с постоянным volume.
- `proxy` — Caddy с авто-HTTPS через Let's Encrypt.
- `backup` — cron `pg_dump` + шифрование + выгрузка off-site.

Секреты в `.env`; статика через WhiteNoise. На деплое: миграции, создание системного администратора, сбор статики. Логирование обязательно; Sentry — опционально.

Для первой версии допустимо держать `web`, `db`, `proxy` и `backup` на одном VPS. Если нагрузка или требования к отказоустойчивости вырастут, БД можно вынести на отдельный сервер/управляемый PostgreSQL, но это не требуется для старта.

## 9. Порядок реализации (фазы)

1. **Каркас:** проект, настройки dev/prod, кастомный `User`, модель `Company`, аутентификация, базовые русскоязычные шаблоны, Tailwind, HTMX.
2. **Ядро тенантности:** `TenantModel`, менеджеры, миксины, middleware, ролевые декораторы/миксины; тесты изоляции.
3. **Справочники и каталог:** `Position` (глобальный, вне тенантности), `Course`, `CourseSession`, `Person`, `CompanyPerson` (CRUD, поиск, локальные дубли, системное сопоставление).
4. **Главный режим — учёт участий:** `Participation`, назначение студентов/ассистентов/тренеров, таймлайн в карточке человека. Это приоритетная фаза.
5. **Вспомогательный режим — прозвон:** `Event` с назначением ассистентов, mobile-first список, `tel:`, инлайн-HTMX отметки `CallRecord` с атомарным захватом против двойного обзвона, фильтры, пагинация.
6. **Безопасность:** ужесточение прав, аудит, согласия/ретенция, экспорт.
7. **Деплой:** Beget Cloud VPS, Docker Compose, HTTPS, бэкапы, мониторинг, демо-данные, приёмочные тесты.
8. **Оплата участия (будущее):** поля оплаты на `Participation` (тип участия, сумма, валюта, дата, заметка), отчёты по оплатам.

## 10. Зафиксированные допущения

- **Главный режим — учёт** участий (студент/ассистент/тренер); прозвон — вспомогательный.
- Изоляция данных между компаниями — построчная, один тенант = одна компания.
- `Person` — глобальная запись личности без `company`; рабочая company-scoped карточка — `CompanyPerson`.
- Один и тот же человек может быть связан с несколькими компаниями без дублирования глобального `Person`.
- Межкомпанийное сопоставление и перепривязка `CompanyPerson` к существующему `Person` доступны только системному администратору.
- Администраторы компаний видят и редактируют только свои `CompanyPerson`; чужие компании и межкомпанийные совпадения им не раскрываются.
- История прозвона — по отдельным встречам (`CallRecord` на каждую `Event`).
- На прозвон назначаются ассистенты; пул общий (не разбит заранее), но человек, которого уже взяли или обзвонили, недоступен другим — через атомарный захват `CallRecord`.
- Оплата участия — **планируемое расширение** на `Participation` (тип участия, сумма и т.д.); модель проектируется с расчётом на это.
- Тренеры — роль участия, не отдельная сущность; единый список людей строится через `Person` + `CompanyPerson`.
- Справочник «Должность» обслуживает поле `chosen_position` в участии.
- Справочник должностей — **глобальный, общий для всех компаний** (вне тенантности; правит системный администратор).
- Полный админ-режим — собственные CRUD-вью; Django Admin — только для системного администратора.
- Весь видимый интерфейс приложения и экспортов — на русском языке; технические enum/code constants могут оставаться на английском.
- Деплой по умолчанию — Beget Cloud VPS + Docker Compose; обычный виртуальный хостинг Beget и one-click Django-образ не являются основным вариантом для этого стека.
- Для первой версии не нужен Celery/Redis: текущий Django + HTMX + PostgreSQL стек достаточен.
